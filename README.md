## 1. 谷歌 *扼杀* JPEG XL？

**原文标题**: Google *Unkills* JPEG XL?

**原文链接**: [https://tonisagrista.com/blog/2025/google-unkills-jpegxl/](https://tonisagrista.com/blog/2025/google-unkills-jpegxl/)

本文探讨了谷歌出人意料地撤销从Chromium中移除JPEG XL支持的决定。尽管最初因认为缺乏兴趣且与AVIF等现有格式相比增益不明显而驳回了该格式，但在社区压力和重大发展推动下，Chromium团队重新考虑了这一决定。

作者指出，到2025年末，JPEG XL的支持仍然仅限于Safari和一些小众浏览器。然而，Chromium对“高性能和内存安全”的JPEG XL解码器的新接受态度预示着该格式可能被广泛采用。

文章回顾了最初拒绝JPEG XL的情况，尽管它收到了来自Meta、Intel和Adobe等主要参与者的积极反馈。作者强调了社区的强烈倡导、Firefox对基于Rust的解码器的兴趣，以及PDF协会采用JPEG XL作为首选图像格式的意图。

作者强烈倡导JPEG XL，列举了其卓越的特性，包括无损JPEG重新压缩、广色域和HDR支持、大图像尺寸支持、高位深和通道支持、对世代损失的抵抗力、渐进式解码、动画支持和Alpha透明度。他总结说，JPEG XL是图像格式的未来，并期望Chromium的支持将是其大规模采用的关键一步。

## 2. 新一轮人工智能寒冬将至

**原文标题**: A New AI Winter Is Coming

**原文链接**: [https://taranis.ie/llms-are-a-failure-a-new-ai-winter-is-coming/](https://taranis.ie/llms-are-a-failure-a-new-ai-winter-is-coming/)

作者认为，尽管基于Transformer的AI模型最初展现出潜力，但由于其根本局限性，一场新的“AI寒冬”正在逼近。Transformer模型虽然克服了一些先前的AI挑战，如扩展性和NP完备性，但它们本质上容易出现“幻觉”——即生成看似合理但实则不正确的输出。

作者解释说，Transformer模型的工作原理是预测序列中最合理的下一个标记，而不管该标记在事实上是否准确或在逻辑上是否合理。这导致了对于许多实际用途来说不可接受的失败率，特别是当更大的模型产生看似令人信服但却是错误的输出，并且只有专家才能识别时。

作者将此与互联网泡沫相提并论，当时过度炒作的技术导致了广泛的企业倒闭和投资者损失。他预测，生成式AI项目，特别是那些基于不切实际期望的项目，将遭受同样的命运，导致像OpenAI这样的公司遭遇重大崩溃，并导致AI基础设施支出回落。

诸如AI辅助编程之类的具体例子，突显了在没有专家监督的情况下依赖Transformer模型的危险，因为细微的错误可能导致严重的问题。作者告诫不要在错误可能造成严重后果的领域（如医疗、执法和教育）中使用Transformer模型。他预计，虽然这项技术不会完全消失，但其应用将仅限于少数“杀手级应用”用例，其余的将会逐渐消失。

## 4. 制图师一直在瑞士地图中隐藏秘密插图。

**原文链接**: [https://eyeondesign.aiga.org/for-decades-cartographers-have-been-hiding-covert-illustrations-inside-of-switzerlands-official-maps/](https://eyeondesign.aiga.org/for-decades-cartographers-have-been-hiding-covert-illustrations-inside-of-switzerlands-official-maps/)

以下是文章“地图制图员一直在瑞士地图中隐藏秘密插图”的简明摘要：

文章详细介绍了瑞士地图制图员在官方瑞士地图中嵌入小型、通常带有幽默感的插图的悠久传统。这些秘密图像，范围从动物和神话生物到日常物品和场景，隐藏在地图的偏远或看似不重要的区域，旨在作为地图制图员之间的一个有趣的内部笑话，并为那些足够细心发现它们的人提供一种微妙的装饰。

这种做法可以追溯到几十年前，并具有多种目的。首先，它可以作为防止版权侵权的保障。如果地图被复制，这些独特的插图的存在可以证明原始作者身份。其次，这是地图制图员在高度技术性和规范性的领域注入个性和创造力的一种方式。最后，它培养了地图制作者之间的社区意识和共享知识。

文章强调，这些隐藏的插图不会干扰地图的主要功能；相反，它们是微妙的补充，为瑞士的地图制图历史增添了一层神秘感和特色。这些图像通常很小，并且融入到景观中，奖励对地图的仔细检查并增加它们的吸引力。文章重点介绍了这些插图的例子，并讨论了渗透到瑞士地图制图中的一丝不苟和注重细节的整体文化。

谷歌的Gemini 3已经展示出具有竞争力的性能，挑战了OpenAI在模型质量上的主导地位，而谷歌潜在的直接出售TPU的行为，则威胁着英伟达在人工智能芯片市场上的高利润率。Thompson质疑哪家公司面临的风险更大，并指出英伟达的盈利能力与OpenAI的财务亏损。

尽管谷歌是一个强大的竞争对手，拥有诸如领先的图像和视频生成等优势，但Thompson认为，改变数百万ChatGPT用户的习惯对谷歌来说是一项重大挑战。

## 8. ImAnim: 为ImGui应用提供现代动画功能

**原文标题**: ImAnim: Modern animation capabilities to ImGui applications

**原文链接**: [https://github.com/soufianekhiat/ImAnim](https://github.com/soufianekhiat/ImAnim)

ImAnim：为 Dear ImGui 应用打造的轻量级动画引擎，旨在帮助开发者以最少的代码添加流畅、现代的 UI 动画。它对即时模式友好，除了 Dear ImGui 之外没有任何依赖，并提供广泛的功能，包括大型缓动函数库（包括弹簧物理）、感知色彩混合以及适应窗口大小调整的响应式布局。

主要功能包括：

*   **补间动画 (Tweens):** 以各种策略（淡入淡出、切割、队列）动画化浮点数、向量、整数和颜色。
*   **片段 (Clips):** 基于时间轴的关键帧动画，具有循环、回调和链接功能。
*   **缓动 (Easing):** 包含预设、三次贝塞尔曲线、步进和弹簧物理的全面缓动函数集合。
*   **运动路径 (Motion Paths):** 沿贝塞尔曲线、Catmull-Rom 样条曲线和沿路径的文本动画元素。
*   **程序动画 (Procedural Animation):** 振荡器、震动、摆动和 Perlin/Simplex 噪声效果。
*   **附加功能 (Extras):** 用于主题过渡的样式插值、滚动动画、调试检查器、文本效果和可感知大小调整的动画。

通过添加两个文件 (im_anim.h 和 im_anim.cpp) 可以轻松地将 ImAnim 集成到项目中。 详尽的文档和一个全面的演示展示了所有功能，包括交互式缓动曲线可视化工具、弹簧物理游乐场以及所有动画类型的实时控件。 该引擎采用 MIT 许可，并通过 Patreon 提供支持。

## 9. 70年代的矢量图形工作站

**原文标题**: A vector graphics workstation from the 70s

**原文链接**: [https://justanotherelectronicsblog.com/?p=1429](https://justanotherelectronicsblog.com/?p=1429)

本文详细介绍了对一台1975年产泰克4051图形工作站的维修和修复过程。作者首先简要介绍了泰克公司的历史，强调了其在示波器领域的先驱地位以及随后利用存储CRT技术进军计算机终端领域的历程。4051由摩托罗拉6800驱动，运行BASIC，在当时提供了先进的矢量图形功能，目标用户是研究人员和分析师。

作者获得了一台状况不明的4051，并记录了维修过程。最初，由于一个有故障的ON/OFF开关和主变压器上一根断开的电线，机器没有反应。修复这些问题后，一个电阻烧毁，需要更换。显示器严重失调，需要使用高压示波器探头调整各种电压水平。

作者的4051具有32KB的RAM和一个ROM扩展器，允许使用多个ROM模块，增加额外的功能。磁带驱动器目前已损坏，但ROM包括一个编辑器、一个二进制数据加载器/保存器，以及对作者没有的软盘驱动器的支持。尽管由于显示技术的原因，游戏选择有限，但有几个演示和程序可用。

未来的计划包括构建一个GBIP闪存仿真器，以便更轻松地加载程序，以及一个用于克隆丢失的ROM卡的ROM板。有关泰克4051的资源、程序和仿真器可以在Monty McGraw的Github页面上找到。

## 10. 自托管 Matrix 服务器五年

**原文标题**: Self-hosting a Matrix server for 5 years

**原文链接**: [https://yaky.dev/2025-11-30-self-hosting-matrix/](https://yaky.dev/2025-11-30-self-hosting-matrix/)

本文详细介绍了作者五年来自行托管 Matrix 服务器的经验，主要使用 Synapse，服务于一小群用户，包括一个 WhatsApp 桥接器。虽然作者赞赏 Matrix 协议的功能以及 Synapse 的可靠性和完善的 API 文档，但他们也指出了几个显著的缺点。

Synapse 缺乏内置的管理面板，需要 PostgreSQL 数据库，而 PostgreSQL 数据库会迅速增长（需要不断清理和 vacuuming），保留废弃的房间和附件，并且不允许删除用户，引发了 GDPR 合规性问题。初始设置假定存在联邦机制，但很难禁用。

本文批评了 Matrix-Element 生态系统的发展方向，特别是新的 Element Server Suite (ESS) Community，认为其 Kubernetes 要求对于小型部署来说过于繁琐。作者更喜欢基于 XMPP 的 Snikket 的简洁性和效率。

作者还评估了 Element 客户端。虽然 Element Classic 提供了跨平台一致性，但它缺少图像标题，并且存在通知缓慢和离线状态指示不明确的问题。Element X 被认为速度较慢，对话排序存在问题，缺乏后台同步，并且需要服务器端的“滑动同步”选项。Element X 上的通话与旧版本不兼容。入门过程，尤其是使用 Element X 时，是一个主要的痛点，使得新用户难以创建简单的用户名/密码帐户。

最终，作者得出结论，以企业为中心的新 ESS 和日益受限的 Synapse/Element X 不适合他们的需求，并计划迁移到 Snikket，因为它具有效率、流畅的入门体验和及时的通知。
