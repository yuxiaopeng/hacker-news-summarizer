## 5. 从GitHub迁移Dillo

**原文标题**: Migrating Dillo from GitHub

**原文链接**: [https://dillo-browser.org/news/migration-from-github/](https://dillo-browser.org/news/migration-from-github/)

罗德里戈·阿里亚斯·马洛正将Dillo项目从GitHub迁移到具有镜像的自托管解决方案，原因是GitHub的局限性和潜在的单点故障。最初的dillo.org站点于2022年丢失，促使需要更具弹性的基础设施。

GitHub的问题包括依赖JavaScript、资源密集、控制权集中、性能缓慢，以及强调“推送模型”通知，这与作者首选的用于离线工作的“拉取模型”不符。还提到了对用户审核以及优先考虑LLM趋势的担忧，这影响了Dillo的兼容性。

自托管解决方案包括dillo-browser.org上的VPS，使用cgit进行Git存储库浏览，以及一个自定义的bug跟踪器“buggy”，该跟踪器利用Markdown文件和Git进行数据存储和版本控制。这确保了数据不会集中化且易于恢复。

Codeberg和Sourcehut上的镜像提供了进一步的冗余。虽然dillo-browser.org的DNS条目仍然是单点故障，但替代通信方法和镜像更新可以缓解其丢失。网站上的OpenPGP签名确保了真实性。

GitHub存储库在迁移期间将保持活动状态，然后会被存档，以避免中断下游构建。作者强调了新基础设施的低成本和低能耗。欢迎通过Liberapay捐款以支持项目的长期可持续性。

## 14. 强化学习比你想象的更信息低效

**原文标题**: RL is more information inefficient than you thought

**原文链接**: [https://www.dwarkesh.com/p/bits-per-sample](https://www.dwarkesh.com/p/bits-per-sample)

这篇博文认为，强化学习（RL）比之前认为的更加信息低效，不仅需要每个样本消耗大量FLOPs。核心论点是，在大多数训练过程中，与监督学习（预训练）相比，RL中每个样本的信息密度（Bits/样本）显著降低。

在监督学习中，每个token都提供直接且信息丰富的信号，纠正模型的预测并最大化学习到的bits。然而，在RL中，尤其是在通过率较低的情况下（模型不太可能成功），反馈信号（成功或失败）提供的信息有限。作者通过比较RL和监督学习中基于通过率的学习信息来证明这一点。RL的信息密度只有在训练的最后阶段，当模型已经表现良好时，才可与监督学习相媲美。

此外，RL在训练初期遭受高方差的影响，导致梯度不稳定。相反，监督学习由于不可约损失，在训练后期面临高方差。

为了提高RL效率，作者建议采用诸如课程学习和自我博弈等策略，使“通过率”保持在“黄金区域”内。代理评估指标也可以提高样本/FLOP密度（更早地估计最终奖励）或Bits/样本密度（即使在失败的尝试中也奖励“良好”的推理）。

尽管信息密度较低，作者承认RL中学习到的bits非常有价值，因为它们直接针对任务性能和学习纠正错误，而不仅仅是吸收数据流形。然而，该文章最后建议，未来的研究应侧重于识别和利用高效人类学习的机制，这些机制可以从环境中提取大量信息。

## 22. HTTP 查询方法

**原文标题**: The HTTP Query Method

**原文链接**: [https://www.ietf.org/archive/id/draft-ietf-httpbis-safe-method-w-body-14.html](https://www.ietf.org/archive/id/draft-ietf-httpbis-safe-method-w-body-14.html)

本文档介绍 HTTP QUERY 方法，旨在弥合 GET 和 POST 请求在执行安全和幂等查询方面的差距。 与 GET 不同，QUERY 允许在请求内容中传递大量数据，而不是在 URI 中。 与 POST 不同，它明确声明了请求的安全和幂等性，从而启用诸如缓存和自动重试等功能。

QUERY 方法指示目标资源处理封闭的内容并返回结果。 服务器必须验证 Content-Type 并拒绝不一致的请求。 QUERY 请求是安全的，意味着它们不会修改目标资源的状态，并且是幂等的，意味着可以安全地重试它们。

该规范定义了 QUERY 请求的“等效资源”的概念，该资源可以通过 GET 访问并代表查询本身，包括内容和元数据。 服务器可以使用 Location 响应头将 URI 分配给这些资源。 Content-Location 标头可以指向查询结果的 URI。

本文档还涵盖了重定向场景，指出与 POST 不同，QUERY 请求在收到 301 或 302 响应后不应重定向到 GET 请求。 条件请求和缓存机制的定义与 GET 类似，但特别考虑了将请求内容纳入缓存键。 范围请求语义也与 GET 的语义相同。

电力供应是首要考虑因素。太阳能发电并不比地面太阳能有显著优势，而为大型数据中心（与OpenAI计划在挪威的设施规模相当）供电所需的规模，需要发射数量庞大、尺寸堪比国际空间站的卫星，这并不现实。核能选项，如放射性同位素热电发生器（RTG），提供的电力甚至不足以支持单个GPU。

作者总结说，虽然技术上可行，但太空数据中心将极其困难，成本高昂，并且与地球上的同类产品相比，性能平庸。

## 50. 加菲尔德的勾股定理证明

**原文标题**: Garfield's Proof of the Pythagorean Theorem

**原文链接**: [https://en.wikipedia.org/wiki/Garfield%27s_proof_of_the_Pythagorean_theorem](https://en.wikipedia.org/wiki/Garfield%27s_proof_of_the_Pythagorean_theorem)

加菲尔德总统的勾股定理证明是美国第20任总统詹姆斯·A·加菲尔德（时任国会议员）于1876年提出的一种独特的定理证明方法。它发表在《新英格兰教育杂志》上，堪称美国总统在数学上的一个非凡贡献。

该证明依赖于用两个全等的直角三角形（ABC和BDE）和一个等腰直角三角形（ABD）构造一个梯形。通过两种不同的方式计算梯形的面积，加菲尔德建立了直角三角形各边之间的关系，从而证明了该定理。

第一种方法使用公式：高 * （平行边平均值）= (a+b) * (1/2)(a+b) 来计算梯形面积。

第二种方法将梯形面积计算为三个三角形面积之和：(1/2)(a*b) + (1/2)(c*c) + (1/2)(a*b)。

通过等同这两个表达式并进行简化，加菲尔德得出了勾股定理：a² + b² = c²。该证明被数学家认为是巧妙的，并被收录在勾股定理证明集中。
