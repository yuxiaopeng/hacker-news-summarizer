## 2. 为什么有些衣服洗后会缩水——以及如何让它们“复原”

**原文标题**: Why some clothes shrink in the wash – and how to 'unshrink' them

**原文链接**: [https://www.swinburne.edu.au/news/2025/08/why-some-clothes-shrink-in-the-wash-and-how-to-unshrink-them/](https://www.swinburne.edu.au/news/2025/08/why-some-clothes-shrink-in-the-wash-and-how-to-unshrink-them/)

这篇文章由纺织科学家尼萨·萨利姆（Nisa Salim）博士撰写，解释了衣物缩水的主要驱动因素是“纤维记忆”。棉和亚麻等天然纤维由天然卷曲的纤维素分子组成。在制造过程中，这些纤维被拉伸并排列成平滑的线条；然而，洗衣机的热量、水分和机械搅拌会导致这些纤维放松，并恢复到原始的卷曲状态。

文章指出，不同材质缩水的原因各不相同：
*   **棉和亚麻：** 它们具有亲水性（趋水）。水分会导致纤维膨胀并变得易于活动，从而在受到搅拌时发生回缩。
*   **羊毛：** 作为一种动物纤维，羊毛表面分布着被称为“角质层”的微小鳞片。在洗涤过程中，这些鳞片会张开并相互锁合——这一过程被称为“毡化”——从而使织物变得更紧密、更小。
*   **合成纤维：** 涤纶和尼龙等材料更耐缩水，因为其石油基聚合物具有稳定的晶体内部“骨架”，可以防止卷曲。

为了防止缩水，作者建议使用冷水、轻柔洗涤模式和低转速脱水。如果衣物已经缩水，通常可以将其浸泡在混有护发素或婴儿洗发水的温水中进行“补救”。护发素中的阳离子表面活性剂能润滑纤维，使其变得足够柔韧，从而可以轻轻拉伸回原状并平铺晾干。虽然这种方法可能无法扭转极度严重的损伤，但通常能恢复足够的尺寸，让衣物可以再次穿用。

## 3. Starlink 漫游 50GB 现已升级为 100GB，达量后限速不限量。

截至2026年1月13日，Starlink已在大多数市场将其“Roam 50GB”套餐免费升级为**Roam 100GB**。

*   **容量翻倍：** 用户现在每月可获得100GB高速流量。
*   **无限低速数据：** 达到100GB限制后，服务将不再中断，而是转为提供无限低速数据。这可以支持电子邮件、短信和通话等基础连接，而流媒体、大文件下载和视频通话等高带宽活动将受到限制。

**区域可用性：**

**通知：**
当用户的高速数据配额使用量达到80%和100%时，Starlink将向用户发送通知。

## 4. 搞懂 API 时的那种令人抓狂的挫败感

**原文标题**: The Unbearable Frustration of Figuring Out APIs

**原文链接**: [https://blog.ar-ms.me/thoughts/translation-cli/](https://blog.ar-ms.me/thoughts/translation-cli/)

本文记录了作者为辅助中文学习，在开发一个简单的 macOS 命令行翻译工具过程中所经历的波折。为了寻找受 API 令牌限制的服务在本地的替代方案，作者尝试使用 Swift 调用 macOS 的原生翻译功能。

这一过程揭示了若干技术障碍：
*   **语言与工具：** 在发现 Zig 不适合调用 Swift 的异步函数后，作者转向了 Swift。他们发现，要调用最新的 Translation API，需要将 `swift-tools-version` 更新至 6.2，并使用 `swift-argument-parser` 库中的 `AsyncParsableCommand` 协议。
*   **API 实现：** `Translation` 框架缺乏针对非 SwiftUI 场景的明确文档。为了实现语言自动检测，作者集成了 `NaturalLanguage` 框架中的 `NLLanguageRecognizer`。
*   **“无法翻译”错误：** 最大的障碍是由于缺失离线语言模型而导致的一个通用错误。作者发现，该 API 使用的模型与系统右键菜单中“翻译”功能所使用的模型是相互独立的。直到通过系统设置手动下载了特定的语言包（简体和繁体中文）后，才最终运行成功。

最终，作者成功创建了该工具，却在事后意识到 macOS Spotlight（`Cmd+Space`）早已提供了他花费数小时试图复现的完全相同的功能。本文幽默地反思了现代 API 集成中那种“难以承受的挫败感”，以及重新发明系统既有功能的讽刺。

## 6. 模拟的边缘：Game Boy 缝纫机 (2020)

**原文标题**: Edge of Emulation: Game Boy Sewing Machines (2020)

**原文链接**: [https://shonumi.github.io/articles/art22.html](https://shonumi.github.io/articles/art22.html)

在《模拟的边缘》（Edge of Emulation）中，作者 Shonumi 记录了一系列兼容 Game Boy 的缝纫机的历史及其模拟过程的技术复杂性。于 2000 年左右发布的 Jaguar JN-100（在美国以 Singer IZEK 1500 之名销售）以及后来的 JN-2000，利用 Game Boy 为数字缝纫和刺绣提供了一个低成本且用户友好的操作界面。

通过通信电缆（Link Cable）连接，Game Boy 充当了机器的主要控制器。诸如《乐乐缝纫机》（Raku x Raku Mishin）和稀有的日本限定软件《马力欧家族》（Mario Family）等作品，让用户能够自主编写花纹、字母以及任天堂主题的刺绣图案。这种合作对消费市场而言具有革命性，它将此前仅限于昂贵工业设备的数字化缝纫技术，以实惠的价格带给了普通消费者。

模拟这些机器的过程是一项耗时数年的挑战。Shonumi 面临着重重障碍，包括实体硬件极高的成本和稀缺性——尤其是 JN-2000 型号和《马力欧家族》卡带。在技术层面上，这些机器比普通的 Game Boy 配件更为复杂。它们使用了一种独特的通信协议，可以在内部和外部时钟之间切换，并与掌机交替扮演“主”与“从”的角色。

此外 nutrition，逆向工程缝纫指令的过程也困难重重。虽然水平 X 坐标使用的是绝对位置，但垂直 Y 坐标遵循的却是一种与机器送布机构相关的非直观逻辑。尽管存在这些障碍，Shonumi 的工作依然保留了一个具有重要文化意义的时代：在那段时期，Game Boy 的功能超越了游戏本身，成为了家庭创作者的实用工具。这些机器见证了这款掌机的多功能性，也见证了它在向大众普及数字刺绣方面所发挥的作用。

## 7. Xoscript 脚本

**原文标题**: Xoscript

**原文链接**: [https://xoscript.com/history.xo](https://xoscript.com/history.xo)

Xoscript 是一种服务器端脚本语言，其历史可追溯至 1993 年，最初是为 Commodore 64 开发的。在 2009 年项目重启并经历数年迁移后，当前版本于 2026 年发布，旨在作为日益复杂的现代脚本环境的轻量化替代方案。

该语言建立在三大核心原则之上：
*   **简单性：** 极简的语法规则，易于学习。
*   **安全性：** 核心关注安全的服务器端实现。
*   **向后兼容性：** 确保跨更新的一致性。

Xoscript 保持着严格的中立与非政治立场。在技术层面，它采用了多种非常规的设计选择，包括无类型数据、动态作用域、受 Smalltalk 启发的“消息传递”系统以及内置容错机制。最终，Xoscript 致力于为寻求精简服务器端工具的开发者提供实用、高效且轻量的体验。
