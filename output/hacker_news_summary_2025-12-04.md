**原文链接**: [https://entropicthoughts.com/transparent-leadership-beats-servant-leadership](https://entropicthoughts.com/transparent-leadership-beats-servant-leadership)

本文反驳了仆人式领导，并提出“透明领导”作为一种更有效的方法。作者认为，仆人式领导，即预测并消除团队成员的障碍，最终是有害的。它会造成依赖性，使领导者负担过重，并且在领导者离开时，团队会毫无准备且脱节。

**原文链接**: [https://javascript.tm/letter](https://javascript.tm/letter)

## 4. 作为一名资深工程师，我不在意聚光灯。

**原文标题**: I ignore the spotlight as a staff engineer

**原文链接**: [https://lalitm.com/software-engineering-outside-the-spotlight/](https://lalitm.com/software-engineering-outside-the-spotlight/)

本文针对大型科技公司（尤其是Staff+工程师）中经常被强调的“追逐聚光灯”的职业道路，提出了一种替代方案。作者作为谷歌的一名高级Staff工程师，将自己在开发者工具和基础设施方面的经验，与Sean Goedecke所描述的以产品为中心的环境进行了对比，后者强调快速迭代和高管的关注度。

作者没有追逐聚光灯，而是提倡**管家式管理**：对一个系统的长期承诺，这可以通过模式匹配和系统性创新来释放复合回报。他们以Bigtrace为例说明了这一点，如果没有他们深刻的背景和长远的眼光，这个项目是不可能实现的。

作者认为，管家式管理能够产生信任，从而提供政治资本，以便在诸如Perfetto中集成AI等短暂趋势会损害精度等核心价值观时，能够说“不”。虽然管家式管理可能会牺牲“高管的关注度”，但它会产生两种替代性的货币：**影子等级**（给依赖你工具的Staff+工程师留下深刻印象）和**效用账本**（跟踪工程健康、重要性、普遍性和规模的指标）。

作者强调，这条道路虽然需要一家盈利且稳定的公司，但由于它优先考虑长期影响和技术所有权而不是炫目的发布，因此常常被忽视。通过专注于构建基础系统，工程师可以在不不断追逐聚光灯的情况下，拥有有意义且具有高度影响力的职业生涯。关键在于从深度、耐心和赋能他人中获得满足感。

## 6. 函数式四叉树

**原文标题**: Functional Quadtrees

**原文链接**: [https://lbjgruppen.com/en/posts/functional-quadtree-clojure](https://lbjgruppen.com/en/posts/functional-quadtree-clojure)

本文介绍了一种使用函数式方法在 Clojure/Clojurescript 中构建四叉树，用于基于 Web 的可视化。四叉树是一种树形数据结构，它能为某些数据区域提供更多细节，同时在其他地方节省资源，使其非常适合根据焦点优化渲染。

作者将直接编辑节点的命令式方法与高效重建整个树的声明式函数式方法进行了对比。其核心思想涉及一种递归算法：检查节点的大小是否基于其与“相机”位置的距离达到了最佳大小；如果不是，则将其分成四个子节点并重复该过程。

本文演示了如何使用 Clojure 的 `postwalk` 和 `prewalk` 进行树的遍历和操作。特别是 prewalk，简化了基于相机位置细分节点的递归模式。

可视化方面涉及使用 atoms 和 watches 创建生成的树与 canvas 元素之间的连接（尽管后来重构为 100% 纯函数式解决方案）。通过哈希每个矩形的中心并将其转换为十六进制颜色，可以实现一致的配色方案。

作者认为，四叉树在资源有限的情况下非常有用，例如在 VR 中，焦点区域需要高分辨率，而外围区域可以缩小。Clojurescript 和 Shadow-cljs 因其简洁性、函数式表达性和易于部署而受到赞扬。核心实现只有大约 25 行代码。

## 8. 费曼大战电脑

**原文标题**: Feynman vs. Computer

**原文链接**: [https://entropicthoughts.com/feynman-vs-computer](https://entropicthoughts.com/feynman-vs-computer)

数值积分的实用性：超越解析方法？

本文认为，出于实际目的，在速度和足够精度方面，使用计算机进行数值积分通常优于费曼技巧等解析方法。作者缺乏高级分析技能，强调只需几行JavaScript代码，就能在不到一秒的时间内对积分进行相当精确的近似计算，利用期望值和随机抽样的概念。

核心思想是将函数积分等同于计算该函数在积分限内随机分布的点上的期望值。本文通过几个例子展示了这一点，并将简单的JavaScript积分函数的结果与已知的解析解进行了比较。

作者随后解决了误差估计问题，引入了一种基于均值标准误差计算近似置信区间的方法。为了提高精度，本文探讨了一种名为“样本单元工程”的技术，其中积分区间被分成若干段，并且样本被更策略性地分配到函数表现出更复杂行为或对积分值有显著贡献的区域。

文章最后承认，数值解可能并不适用于所有情况，特别是当积分结果是一个函数而不是一个数值时。然而，在许多实际场景中，基于计算机的数值积分的速度和足够的精度使其成为手动解析方法更可取的替代方案。

## 16. 什么是“开放递归”？

**原文标题**: What Is "Open Recursion"?

**原文链接**: [https://journal.stuffwithstuff.com/2013/08/26/what-is-open-recursion/](https://journal.stuffwithstuff.com/2013/08/26/what-is-open-recursion/)

本文解释了“开放递归”的概念，阐述了如何在仅有函数（lambda表达式）和记录（如JavaScript对象字面量）的更简单语言中构建面向对象特性。作者使用Dart作为lambda演算的替代品来阐释这个概念。

简单函数映射和真正对象之间的关键区别在于，对象方法可以相互调用（递归），并且可以在派生类中被重写（开放性）。

本文演示了如何使用Dart映射和闭包创建一个计数器“对象”。最初的尝试失败了，因为这些方法（increment、get、set）不能相互调用，因为它们不是同时定义的。这个问题通过使用函数表达式和变量重新赋值来创建相互递归的定义来解决。

然而，这仍然没有实现真正的面向对象行为。当试图创建一个扩展基本计数器的“日志计数器”时，被重写的方法会被隐藏，而不是被正确重写。基本计数器的`increment`方法使用原始的`get`和`set`，而不尊重日志记录版本。

为了实现虚方法（从而实现正确的继承），本文展示了如何将接收器对象（类似于`this`或`self`）显式地传递到基本计数器函数中。这“打开”了基本方法的闭包，允许它们查看并调用派生对象中的重写方法。本质上，基本方法不再封闭彼此的变量，而是引用接收器来启用多态行为。

因此，“开放递归”意味着方法是相互递归的，并且可以访问派生接收器对象，从而实现继承和重写。

## 36. WinQuake 存在的原因及工作原理

**原文标题**: Why WinQuake exists and how it works

**原文链接**: [https://fabiensanglard.net/winquake/index.html](https://fabiensanglard.net/winquake/index.html)

Fabien Sanglard 的文章深入探讨了 WinQuake 存在的理由及其内部运作机制，并将其与原始的 quake.exe 进行了对比。作者阐明，WinQuake 的出现是为了解决在 Windows 95 虚拟机中运行 quake.exe 时遇到的性能问题，与直接在 DOS 中运行相比，这会产生显著的开销。此外，WinQuake 提供了对 Windows TCP/IP 堆栈的直接访问，并实现了与 Windows NT 的兼容性，而原始的基于 DOS 的 Quake 在这方面存在困难。

文章探讨了 WinQuake 的配置选项（wq max, fast 等）及其相应的性能基准，揭示了输入、音频和视频的各种后端。 WinQuake 使用 DirectSound 或较旧的 WinMM API 进行音频处理，使用 DirectInput 或 winuser.h 进行输入处理，并为视频提供了多个渲染路径，包括 GDI、VGA、VESA、加速 VESA 和 DirectDraw，这些路径由 SciTech 的 MegaGraph Graphics Library (MGL) 管理。

作者强调了 MGL 在当时复杂的视频系统领域中导航的重要性，它提供了对 VBEAF、VBE、DirectDraw 和标准 VGA 模式的访问。WinDirect 是 SciTech 的一项专有技术，它使 WinQuake 能够绕过 GDI/DirectDraw 以实现直接硬件访问。DirectDraw 提供了优于 GDI 的性能改进，但其可靠性取决于驱动程序兼容性和 DirectX 安装。文章总结说，WinQuake 成功地优化了渲染性能，为声音提供了后端选择，并提供了优于 quake.exe 的体验，从而证明了其开发的合理性，并确保了即使在现代 Windows 系统上也能持久运行。

## 63. C语言中的定长数组形参

**原文标题**: Checked-size array parameters in C

**原文链接**: [https://lwn.net/SubscriberLink/1046840/3eb9029084cc9e1e/](https://lwn.net/SubscriberLink/1046840/3eb9029084cc9e1e/)

LWN.net文章探讨C语言中带大小检查的数组参数用法，尤其是在Linux内核语境下。文章指出一个潜在的安全问题：即使函数原型中指定了大小，编译器也并非总是检查传递给函数的数组参数的大小。

文章介绍了两种解决此问题的方法：

1.  **指针间接引用:** 修改函数原型，使用指向特定大小数组的指针。这迫使调用者提供指向正确大小数组的指针，从而启用编译时的大小检查。
2.  **`static`关键字:** 在数组参数声明中使用`static`关键字(例如，`u8 nonce[static XCHACHA20POLY1305_NONCE_SIZE]`)。这告诉编译器，传递的数组必须至少具有指定的大小，如果更小，则会生成警告。它不需要在调用点进行更改。

尽管`static`方法因其语法而被认为是“可怕的技巧”，但它已在内核的部分代码中使用。文章提到，即使没有`static`，GCC也可能生成“数组太小”的警告，但这些警告目前在内核中被禁用。然而，使用`static`可能会在Clang中启用这些警告。

评论中的讨论表明，`static`关键字方法的有用性存在争议，因为一些评论员认为它没有提供任何优于标准数组声明（例如，`int a[n]`）的好处，甚至可能引入潜在的问题。他们声称，现代编译器（如GCC）已经通过适当的标志提供了数组边界检查，从而使`static`变得不必要。
