## 10. 光谱渲染，第二部分：实时渲染

**原文标题**: Spectral rendering, part 2: Real-time rendering

**原文链接**: [https://momentsingraphics.de/SpectralRendering2Rendering.html](https://momentsingraphics.de/SpectralRendering2Rendering.html)

光谱渲染，第二部分：实时渲染

本文探讨了将光谱渲染集成到实时应用中的高效方法，重点是通过蒙特卡洛积分和重要性采样来评估光谱上的积分。它建立在第一部分的基础上，利用光源和反射光谱来确定像素颜色。

作者详细介绍了如何使用蒙特卡洛方法，基于光源光谱和颜色匹配函数，有效地计算颜色积分。与采样固定数量的波长相比，这种方法旨在最大限度地减少计算开销。本文解释了通过反向累积分布函数采样和受 Hero 波长采样启发的均匀抖动采样来实施波长采样的方法，并利用预计算的查找表来优化性能。

本文还讨论了 BRDF 对光谱渲染的适应性，特别是以 Frostbite BRDF 为例，其中颜色表示为基色和纯白色的权重，从而实现光谱反射率的计算。一个关键问题是由于蒙特卡洛积分引入的颜色噪声，文章比较了在不同光源下 RGB 和光谱渲染之间的噪声水平。结果表明，通过良好重要性采样进行光谱渲染产生的颜色噪声通常可以忽略不计，与其他蒙特卡洛噪声源相比，现有的降噪器可以有效地缓解它。

作者承认了处理多个光源的挑战，并提出了针对直接光照和较长路径场景的策略。他们简要提到了引导方法，作为在渲染过程早期确定相关光源的潜在解决方案，为在复杂场景中实现更高效的光谱渲染铺平了道路。

## 13. Intel 386标准单元逻辑中的异常电路

**原文标题**: Unusual circuits in the Intel 386's standard cell logic

**原文链接**: [https://www.righto.com/2025/11/unusual-386-standard-cell-circuits.html](https://www.righto.com/2025/11/unusual-386-standard-cell-circuits.html)

本文深入探讨了英特尔386处理器标准单元逻辑中不寻常的电路设计。面对386的复杂性，英特尔采用了标准单元逻辑，通过使用预先设计的单元（如反相器和与非门）来自动化布局。这种方法虽然有风险，但加快了开发速度。

作者重点介绍了三个出乎意料的电路：超大型多路复用器、错位的晶体管和“假”反相器。由于x86架构的寄存器灵活性，寄存器控制逻辑使用了复杂的多路复用器。这些多路复用器使用CMOS开关和反相器进行放大，需要很大的面积。

错位的晶体管位于标准单元列之外，据推测是为了快速修复错误，避免代价高昂的单元再生。由于这种异常，作者的分析工具最初难以正常工作。

最后，发现一些反相器是“假”的。虽然物理上与反相器相同，但它们的输入是分开的，允许它们充当独立的NMOS和PMOS晶体管。这些晶体管被纳入多路复用器逻辑中，增加了更多的选择情况。

本文强调了386设计中为赶上截止日期而采用的独创性和潜在的“黑客”技术，突出了标准单元逻辑中隐藏的复杂性和惊喜。

## 14. 几乎所有考拉兹轨道都达到几乎有界的值

**原文标题**: Almost all Collatz orbits attain almost bounded values

**原文链接**: [https://mathvideos.org/2023/terence-tao-almost-all-collatz-orbits-attain-almost-bounded-values/](https://mathvideos.org/2023/terence-tao-almost-all-collatz-orbits-attain-almost-bounded-values/)

本文总结了陶哲轩在考拉兹猜想上的工作。考拉兹猜想是数学中一个著名的未解问题，它假设对于任何自然数*n*，重复应用考拉兹映射（如果*n*是奇数，则为3*n*+1；如果*n*是偶数，则为*n*/2）最终将导致1。

陶哲轩的工作并没有解决这个猜想，但通过证明“几乎所有”考拉兹轨道都达到“几乎有界”的值，从而显著提升了我们对它的理解。具体来说，他证明了对于任何趋于无穷大的函数*f(n)*（无论多么缓慢），几乎所有考拉兹序列*n*，Col(*n*)，Col²(*n*)，…最终都会达到一个小于*f(n)*的值。

陶哲轩成果的关键在于构建了一个用于考拉兹动力学的“近似不变”或“自相似”测度，该测度在加速过程后生效。这种测度允许对轨道进行概率分析，从而深入了解它们的长期行为，并得出关于几乎所有轨道都达到几乎有界值的结论。该视频是高等研究院座谈会的一部分。

## 32. `satisfies` 是我最喜欢的 TypeScript 关键字 (2024)

**原文标题**: `satisfies` is my favorite TypeScript keyword (2024)

**原文链接**: [https://sjer.red/blog/2024-12-21/](https://sjer.red/blog/2024-12-21/)

本文探讨了 TypeScript 关键字 `satisfies`，以及作者认为其是一种有价值工具的原因。作者首先解释了 TypeScript 并非总是能推断出最精确的类型，从而导致潜在的类型不匹配，尤其是在函数期望特定字面量类型时。

传统的类型注解可以解决这个问题，但会强制指定类型，有时会妨碍 TypeScript 基于提供的值推断出更具体信息的能力。

`satisfies` 关键字提供了一种替代方案。它确保一个值可以赋值给指定的类型，但允许 TypeScript 基于值本身推断出更具体的类型。这类似于类型安全的类型转换。

作者用一个 `Person` 类型示例说明了这一点。使用 `satisfies`，可以将变量声明为 `Person`，但 TypeScript 会推断字段的具体值（例如，`name` 为 `"Jerred"`，`isCool` 为 `true`），而不是仅仅推断为 `string` 或 `boolean`。这使得变量可以传递给期望更具体类型的函数，而无需显式类型注解，这些注解可能会模糊具体值。

本质上，`satisfies` 允许进行类型检查，同时保留 TypeScript 的类型推断能力，从而在需要精确类型信息的某些情况下提供了一种更灵活和强大的方法。

作者展示了两个案例研究：“The Print Shop 1.0”，它需要正确的开放总线仿真才能运行；以及“Animation Toolkit 1.0”，它因一个依赖于特定开放总线行为的 bug 而崩溃。文章强调了准确模拟硬件行为的重要性，即使它导致崩溃，也能实现最大的保真度。作者强调，bug 级别的精确仿真通常是最佳方法。
