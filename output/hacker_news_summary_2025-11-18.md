## 8. Strix Halo的内存子系统：应对iGPU挑战

**原文标题**: Strix Halo's Memory Subsystem: Tackling iGPU Challenges

**原文链接**: [https://chipsandcheese.com/p/strix-halos-memory-subsystem-tackling](https://chipsandcheese.com/p/strix-halos-memory-subsystem-tackling)

AMD Strix Halo APU内存子系统探究：移动环境下的CPU与iGPU性能平衡

本文深入探讨了AMD Strix Halo APU的内存子系统，重点关注移动环境下平衡CPU和集成GPU (iGPU) 需求的挑战。 Strix Halo为iGPU配备了32MB的Infinity Cache，类似于RX 7600的架构，以缓解主内存的带宽压力。 虽然Infinity Cache有所帮助，但其延迟略高于独立GPU的实现。 文章指出，Infinity Cache的行为由软件动态控制，可能通过驱动程序更新进行调整。

Strix Halo的CPU端采用16个Zen 5核心，分布在两个CCD上，通过台积电的InFO_oS封装连接。 虽然与桌面级产品相比，CCD在芯片边界具有更高的写入带宽，但由于大多数工作负载的读取密集型特性，其影响微乎其微。 跨CCX延迟比桌面系统高得令人惊讶，这可能会抵消先进封装带来的任何优势。

文章强调了CPU和iGPU之间共享内存的挑战，导致潜在的争用和在GPU重负载下CPU延迟的增加。 微基准测试表明，增加的GPU带宽需求会显著影响CPU内存延迟。 性能监控单元 (PMU) 数据强化了Strix Halo上高CPU缓存命中率的重要性，原因是潜在的带宽限制。 虽然仅CPU工作负载显示对延迟的影响最小，但使用iGPU进行游戏会显示出显著的增加。

最终，Strix Halo的内存子系统在移动功耗范围内平衡CPU和iGPU的性能需求方面面临着独特的挑战。

## 13. Quake.exe 如何获得它的 TCP/IP 协议栈

**原文标题**: How Quake.exe got its TCP/IP stack

**原文链接**: [https://fabiensanglard.net/quake_chunnel/index.html](https://fabiensanglard.net/quake_chunnel/index.html)

本文深入探讨了1996年发布的《雷神之锤》如何在DOS和Windows 95之间实现跨平台兼容的技术独创性，特别是关注其TCP/IP网络功能。

最初，id Software使用DJGPP编译器（GCC的一个移植版本）创建了一个单一的可执行文件`quake.exe`，该文件与两种操作系统都兼容。这是通过利用DJGPP的DPMI客户端实现的，该客户端可以在DOS和Windows 95 DPMI服务器上运行，从而有效地弥合了这两个环境之间的差距。

对于多人游戏，DOS版《雷神之锤》主要依赖于IPX，需要一个DOS TSR。TCP/IP由于DOS中缺少内置协议栈而难以实现，并且需要昂贵的TSR。

Windows 95提出了不同的挑战和机遇。虽然`quake.exe`可以在DOS-box中运行，但通过使用`q95.bat`的第三种方法解锁了Windows 95的TCP/IP协议栈。这是通过Mpath Interactive的“Chunnel”技术实现的，id Software获得了该技术的许可。

“Chunnel”由几个组件组成：`qlaunch.exe`、`quakeudp.dll`、`genvxd.dll`和`mgenvxd.vxd`。`quakeudp.dll`充当了连接到微软TCP/UDP/IP协议栈(`wsock32.dll`)的桥梁。一个虚拟设备驱动程序`GENVXD.VXD`通过中断0x48与win32通信。Mpath提供了BSD socket函数的代码。这个复杂的系统允许DOS可执行文件使用Windows 95的网络。

当id转向win32独占二进制文件（如`winquake.exe`）并直接访问`wsock32.dll`时，这种复杂的设置就变得过时了。

## 14. 不需要就不要把你的网站放在Cloudflare后面

本文反对为那些并不真正需要的网站使用Cloudflare，强调了依赖中心化服务的风险。作者以最近一次Cloudflare宕机导致大量网站（包括流量很小的网站）瘫痪为例，说明了这种单点故障的弊端。

## 19. 实验：让 TypeScript 默认不可变

**原文标题**: Experiment: Making TypeScript Immutable-by-Default

**原文链接**: [https://evanhahn.com/typescript-immutability-experiment/](https://evanhahn.com/typescript-immutability-experiment/)

Evan Hahn 尝试使 TypeScript 默认不可变，效仿 Rust 等语言的行为。目标是完全通过 TypeScript 配置实现这一点，不依赖于 lint 规则或外部工具。

过程包括：

1. **移除默认库：** 在 `tsconfig.json` 中使用 `noLib` 标志禁用 TypeScript 的内置类型定义，强制创建自定义替代项。
2. **创建最小标准库：** 在 `lib.d.ts` 文件中定义 `Array`、`String` 和 `Object` 等基本接口，以避免移除默认库后出现编译错误。
3. **使数组不可变：** 使用 `readonly` 属性重新定义 `Array` 接口，并排除 `push` 等方法，以默认强制实施不可变性。创建了一个 `MutableArray` 接口，以便在明确指定时允许可变数组。
4. **扩展到 Records：** 对 `Record` 实用程序类型应用类似的策略，创建不可变 (`Record`) 和可变 (`MutableRecord`) 版本。

该实验通过自定义类型定义成功地实现了数组和 Records 的默认不可变行为。然而，作者在使普通的 JavaScript 对象（例如，`const obj = { foo: "bar" }`）默认不可变方面遇到了困难，未能找到合适的类型定义来阻止属性突变。尽管取得了部分成功，作者还是邀请大家提供使所有 TypeScript 对象默认不可变的解决方案。

**原文链接**: [https://blog.cloudflare.com/so-long-and-thanks-for-all-the-fish-how-to-escape-the-linux-networking-stack/](https://blog.cloudflare.com/so-long-and-thanks-for-all-the-fish-how-to-escape-the-linux-networking-stack/)

Chris Branch's article, "How to Escape the Linux Networking Stack," details Cloudflare's journey to overcome limitations in the Linux networking stack while implementing soft-unicast, a method for sharing IP addresses between machines. The core problem revolves around ensuring that TCP connections using soft-unicast addresses don't collide with other processes binding to the same addresses, especially when using Netfilter (iptables) for source NAT.

The initial approach of relying on Netfilter's conntrack caused issues where connections could be silently rewritten to invalid ports, disrupting soft-unicast functionality. Moving to a solution of terminating and proxying connections via local sockets was viable but introduced resource overhead and latency.

The article explores several attempts to reconcile packet rewriting and bound sockets. One involved using Netlink to manipulate conntrack entries, which proved inefficient due to the overhead of Netlink and the need to manually manage connection timeouts. A more promising solution leverages the `TCP_REPAIR` and `TCP_FASTOPEN` socket options to create "connected" sockets without a full handshake, preventing other processes from binding to the same addresses.

However, even this faced challenges. Packets intended for forwarding were being routed locally due to Linux's routing rules. This was addressed by manipulating routing rules to prioritize forwarding marked packets. Finally, early demux, an optimization that looks up local sockets early in packet processing, interfered with the intended routing. The solution involved disabling `tcp_early_demux`, followed by performance evaluation.

## 99. AirPods libreated from Apple's ecosystem

**原文标题**: AirPods libreated from Apple's ecosystem

**原文链接**: [https://github.com/kavishdevar/librepods](https://github.com/kavishdevar/librepods)

LibrePods is an open-source project aiming to unlock the full potential of Apple AirPods on non-Apple devices, specifically Android. It provides access to features normally locked within the Apple ecosystem, such as noise control modes, adaptive transparency, ear detection, battery status, and hearing aid customization.

The software is primarily tested and fully supported on AirPods Pro (2nd and 3rd Gen) and AirPods Max. Basic features should work on other AirPods models. Key features include noise control, automatic music pausing with ear detection, accurate battery readings, head gesture call answering, conversational awareness, hearing aid configuration, and multi-device connectivity (up to two devices). Some advanced features like transparency mode customization and hearing aid require enabling the Bluetooth DID (Device Identification) hook, which simulates an Apple device.

On Android, a rooted device with the Xposed framework is generally required due to limitations in the Android Bluetooth stack, although ColorOS/OxygenOS 16 users can access basic features without root. Enabling the DID hook grants access to multi-device connectivity and accessibility settings like customized transparency and hearing aid configurations. Users can also rename their AirPods and customize long-press actions.

The project is under development for Linux as well. The project is licensed under the GNU General Public License v3.
